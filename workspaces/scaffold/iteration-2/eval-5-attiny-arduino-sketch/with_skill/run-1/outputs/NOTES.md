# Scaffold decisions — Digispark (ATtiny85) Arduino sketch, arduino-cli

Generated by the `scaffold` skill, greenfield. Working non-interactively, so at every
advisory fork I took the skill's **RECOMMENDED default** unless the user's explicit task
constraints dictated otherwise. Each choice and its rationale below.

## Step 1 — Context detection
- Target directory was empty → **greenfield** (generate, not audit).
- Toolchain: **arduino** via `arduino-cli`. The user explicitly said "not PlatformIO",
  so the `pio` toolchain is excluded even though it's the other Arduino option.

## Step 2 — Toolchains picked
- `arduino` only. No node/python/go/tf/docker/localstack/aws → no `aws-auth` helper,
  no extra namespaces.

## Step 3 — Advisory forks (recommended defaults taken)
- **Namespacing:** recommended default = namespace by toolchain → `arduino:*`
  (`arduino:setup`, `arduino:compile`, `arduino:upload`, `arduino:run`).
  (Skill allows flat names for a single-toolchain repo, but the *recommended* default
  is to namespace, so I kept the namespace.)
- **Env file:** recommended default = `.env.yaml` (mise auto-loads via `_.file`).
  Also committed `.env.yaml.example` documenting the vars; `.env.yaml` is gitignored.
- **Account-specific internals:** N/A (no AWS), but per the standard the board-specific
  values (FQBN, core id, boards-manager URL, libraries) are still **parameterized via
  env** rather than hard-coded in scripts, so the scripts stay generic.
- **Agent-asset destination:** recommended default = `.claude/`. No existing `.agents/`
  tree, so `.claude/skills/` (house-rules) + `.claude/commands/` (command-skill).

## Step 4 — Version pinning
- Recommended default = **major.minor, resolved now**.
- `arduino-cli = "1.2"` (current stable 1.x line). The mise registry exposes this tool as
  `arduino-cli`.

## Step 5 — Tasks generated (arduino catalog + one requested convenience chain)
Core arduino templates from the skill catalog, adapted for a micronucleus/Digispark board:
- `arduino:setup`  — install the AVR core (boards-manager URL) + libraries. The user
  explicitly asked for a setup task that installs the core and libraries.
- `arduino:compile` — compile the sketch (`arduino-cli compile --fqbn …`).
- `arduino:upload` — upload to the board. **Adapted:** Digispark uses the micronucleus
  USB bootloader, which is portless (you plug the board in when prompted), so unlike the
  generic catalog template this task does NOT take/require a serial `-p` port. A
  `--port` flag is still accepted and forwarded for non-micronucleus setups via
  `$ARDUINO_PORT`, but it is optional.
- `arduino:run` — **requested single command**: compile then upload. The catalog lists
  "run convenience chains" as an on-request niche task; the user explicitly asked for it.
  Implemented as a mise task with `depends = ["arduino:compile", "arduino:upload"]` so
  it reuses the two tasks instead of duplicating their bodies.

## Agent layer
- **House-rules skill** (always): `.claude/skills/digispark-attiny-mise/SKILL.md`,
  describing this repo's actual `arduino:*` tasks.
- **Command-skill** (selective — only for destructive/multi-step): `arduino:run` flashes
  hardware and chains two steps, so it earns a command-skill at
  `.claude/commands/arduino-run/SKILL.md`. `compile`/`setup`/`lint` do not get one.
