# Scaffold decisions

Generated by the `scaffold` skill. No user present, so at every advisory fork the skill's **recommended
default** was taken. Recorded here.

## Context

- **Greenfield** — target directory was empty.
- Task: new repo for AWS infrastructure as Terraform; one module per region under `modules/` (`global`,
  `sa-east-1`, `us-east-1`); standard init/plan/apply/fmt/check flow taking a module arg; EKS deploys, so
  `kubectl` + `helm` must be available; apply must be careful (show a plan, confirm before changing anything).

## Toolchains selected

- **terraform** — core, the whole point of the repo.
- **aws-auth** — included because this is AWS infra; other tasks source it as a helper.
- **kubectl** and **helm** — pinned in `[tools]` so they are on `PATH` for EKS work. No dedicated tasks were
  generated for them: the request was only that they "be on hand," and the EKS deploy steps are not yet
  specified, so generating speculative `kubectl`/`helm` tasks would violate the skill's "keep it lean / cover
  the repeated loop, not every conceivable operation" guidance. `aws-auth` exposes an `eks_update_kubeconfig`
  helper for when those deploys are wired up.

## Advisory forks — all defaults taken

| Fork | Default chosen | Note |
|------|----------------|------|
| Namespacing | Namespace by toolchain (`tf:*`) | Polyglot-ready and gives `**:` fan-out; one namespace today (`tf`) but kubectl/helm could add more. |
| Env file | `.env.yaml` (mise `_.file`) | House default; committed `.env.example`, gitignored `.env.yaml`. |
| Version pinning | major.minor resolved now | `terraform = "1.13"`, `aws-cli = "2"`, `kubectl = "1.32"`, `helm = "3.16"`. |
| Account internals | Parameterized via env | Region/cluster from `.env.yaml`; nothing account-specific in scripts. |
| Agent-asset destination | `.claude/` | No existing `.agents/` tree, so the default. |

## Tasks generated (`tf` namespace)

`tf:init <module>`, `tf:plan <module>`, `tf:apply <module> [--auto-approve]`, `tf:fmt`,
`tf:check <module>` — the exact init/plan/apply/fmt/check flow requested, each taking the module arg
(except `fmt`, which is repo-wide).

### Careful apply

The skill's catalog `tf:apply` template applies directly unless `--auto-approve`. The user explicitly asked
for a careful apply that "show[s] a plan and confirm[s] before it changes anything," so `tf:apply` was
hardened beyond the template: it writes a saved plan, prints it, requires an explicit `yes`, then applies
**that saved plan** (so what was reviewed is exactly what is applied). `--auto-approve` still bypasses for CI.

## Agent layer

- **House-rules skill** (always): `.claude/skills/aws-infra-mise/SKILL.md` — teaches future agents to use
  `mise run tf:*` and the EKS auth helper instead of raw `terraform`/`kubectl`/`helm`.
- **Command-skills** (destructive/multi-step only): `tf-apply` (changes live AWS state) and `tf-plan`
  (the review step that precedes it). `fmt`/`check`/`init` are not wrapped — they need no guardrails.

## Module placeholders

Each region module got a minimal `main.tf` (terraform block + AWS provider for that region) so the modules
exist and `tf:fmt`/`tf:check` have something real to act on. Replace with actual resources.
